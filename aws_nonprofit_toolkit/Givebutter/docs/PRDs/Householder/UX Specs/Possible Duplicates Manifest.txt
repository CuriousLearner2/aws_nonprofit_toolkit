# DonorTrust / Householder (v1) Project Brief

**Status:** v1 Specification & Design Roadmap (Audit-Safe, Human-in-the-loop)
**Platform:** Desktop Web Interface
**Goal:** Provide a non-destructive interface for reviewing Givebutter CSV imports, identifying duplicate contacts, and normalizing data without altering raw source records.

## 1. Product Vision & Core Principle
DonorTrust (Householder) is a specialized data processing tool designed to bridge the gap between raw CSV exports and clean, audit-ready donor data. 

**The Golden Rule of v1:**
> “The system suggests. The reviewer decides. Raw data stays unchanged.”

### Core Design Principles
- **Non-Destructive:** Marking a record as a duplicate or normalizing an email does NOT merge contacts, overwrite raw CSV rows, or change Givebutter source data.
- **Audit-Safe:** Every reviewer decision — approve, reject, mark same person, mark different, or defer — is logged with a timestamp and reviewer ID.
- **Data Transparency:** The interface must clearly separate "Raw Import Data" from "System Suggestions."
- **Desktop Efficiency:** Optimized for large-screen data density, side-by-side comparisons, and keyboard-efficient navigation.

### v1 Safety Model
Householder v1 processes only the **currently uploaded** Givebutter CSV/import. It is strictly non-destructive.

**v1 does NOT include:**
- Cross-import matching
- Lifetime giving
- Master people
- Global households
- Prior import history
- Identity links
- Givebutter writeback
- Automatic merges
- Automatic approvals

## 3. Finalized Screen: Duplicate Candidate Review
**Screen Title:** “Possible Duplicate in This Import”

### Purpose
Review whether two contact records from the same uploaded CSV appear to represent the same person.

### Core Layout & UX
- **Contextual Header:** Clear match headline with confidence score and summary strip.
- **Side-by-Side Cards:** Contact A vs. Contact B comparison.
- **Evidence Panels:** Explicit lists for "Supporting Evidence" (e.g., exact email) and "Conflicting Evidence" (e.g., address differs).
- **Mandatory Accountability:** Required reviewer reasoning field when conflicting evidence exists.
- **Action Group:** "Mark as Same Person" (Positive), "Mark as Different" (Negative), and "Defer" (Neutral).
- **Details Drawer:** Right-side persistent drawer for Duplicate Candidate Details.

### Safety Enforcement
- **Mark as Same Person** records a reviewer decision only.
- It does **not** merge contacts or donation transactions.
- It does **not** overwrite raw data or change original CSV rows.
- It does **not** write back to Givebutter.

## 5. Success Metrics
- **Reviewer Speed:** Time to resolve a straightforward duplicate candidate (Target: < 5 seconds). Conflicting-evidence cases may take longer and require notes.
- **Audit Integrity:** 100% of reviewer decisions captured in the Audit Log; notes required for conflicting or high-risk decisions.
- **Zero Data Loss:** Absolute preservation of raw CSV rows during the processing lifecycle.

## 7. Project Glossary
To maintain an audit-safe and operator-friendly tone, use the following terminology:

The ‘Avoid’ terms are not merely wording preferences. They describe concepts that are out of scope for v1 and must not appear in UI behavior, routes, data models, or backend logic.

- **Use:** Review, Suggestion, Reviewer Decision, Current Import, Uploaded CSV, Audit Trail, Approved Export.
- **Avoid:** Auto-resolve, Master person, Global identity, Global household, Identity link, Lifetime donor profile, Lifetime giving, Automated merge, Cross-import intelligence.
